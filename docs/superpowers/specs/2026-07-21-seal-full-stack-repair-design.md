# Seal 全栈修复与体验优化设计

**日期：** 2026-07-21
**状态：** 待书面确认
**目标：** 在保留现有 SwiftUI、Core Data、签名服务和安装通道主体架构的前提下，系统修复安全、事务一致性、签名安装状态、前端交互、主题、无障碍及测试流水线问题。

## 1. 范围和成功标准

本次工作覆盖当前审查中发现的前后端问题，不做无关的产品功能扩张，也不把项目整体重写。

完成标准：

- 配对记录、证书、Bundle ID 和账号团队选择不会发生未经用户同意的敏感操作。
- IPA 文件、签名工作区与 Core Data 写入具备可回滚的一致性，失败后不丢失旧文件、不产生“幽灵应用”。
- 导入、签名、安装、批量续签、自签替换与清理任务使用明确且可恢复的状态机。
- 所有长任务都有可见取消入口；底层暂时不可中断时，界面明确显示“正在等待当前步骤结束”。
- 前端不再强制深色：浅色为柔和蓝灰，深色为深海军蓝，均通过语义颜色自动适配系统主题。
- 关键页面支持小屏、横屏、Dynamic Type 和 VoiceOver。
- 单元测试、UI 测试和 CI 与实际产品行为一致，并覆盖本次高风险修复。

## 2. 总体实现策略

采用“保留边界、收紧契约、分阶段替换”的方式：

1. 先用失败测试固定安全与事务缺陷。
2. 在基础设施层引入小型事务和操作协调抽象，不把状态补丁散落到视图中。
3. 修正签名、安装和续签状态机，再接入前端取消与恢复入口。
4. 最后统一主题、布局、无障碍和加载性能。
5. 每阶段独立构建和测试，避免把多个故障合并成难以定位的大改动。

## 3. 后端与基础设施设计

### 3.1 配对记录安全

- 传统配对路径不再长期启动能够响应任意本地客户端 `ReadPairRecord` 的明文 TCP 服务。
- 优先把凭据限制在当前安装操作的进程内和最短生命周期内，并在操作结束后清零缓存、关闭监听。
- 如果现有底层库无法在不公开私钥的情况下消费旧格式配对记录，则安全地拒绝该旧路径并提示用户重新配对；不继续保留可窃取私钥的兼容降级。
- 新增测试验证未授权请求不能读取配对 XML，取消或失败后监听与敏感缓存均被释放。

### 3.2 Apple Portal 安全策略

- 达到证书数量上限时只返回可理解的容量错误与证书信息，不自动撤销任何远端证书。
- 默认映射 Bundle ID 为原标识加 `.seal`，降低命中正式 App ID 的概率。
- 复用已存在的 Portal App ID 时对 capabilities 与 entitlements 做保守合并，不用本地过滤结果覆盖远端已有配置。
- 登录后若存在多个团队，必须让用户明确选择；所选团队随账号记录保存，后续签名不得静默切换。

### 3.3 文件与数据库事务

引入两阶段文件提交结果：

```text
stage 新文件
  -> prepare（旧文件移入可恢复备份，新文件就位）
      -> Core Data 成功 -> finalize（删除备份）
      -> Core Data 失败 -> rollback（恢复旧文件并移除新文件）
```

- 导入、自签替换和缓存清理共用同一事务规则。
- Core Data 删除失败必须 rollback context。
- 重复记录清理同时处理受控文件，恢复扫描不会把失败残留重新登记为已安装应用。
- 孤儿 IPA 只能恢复为“已导入/待检查”，不能凭文件存在推断为已安装。

### 3.4 操作协调与清理

- 增加共享操作协调器，为导入、签名、安装、续签、自签替换和清理提供应用级 lease。
- 清理操作跳过正在使用的应用和工作区；已安装身份与当前瞬时进度分开判断，避免签名中状态暂变后被当成缓存删除。
- 设置页的破坏性操作显示 busy 状态，禁止重复触发，并在失败后保留可重试信息。

## 4. 签名、安装与续签状态机

- 接入现有但未使用的已签名 IPA 缓存路径：验证文件存在且元数据匹配后才能复用。
- 取消采用结构化任务传递 `CancellationError`，不把取消包装成普通失败。
- 对确实不可中断的同步安装调用，在调用前后检查取消，并向 UI 暴露“取消已请求”的中间状态。
- 导入预检完成后仍停留在确认界面，不再自动确认和关闭。
- 自签替换保留原账号、团队、安装时间、到期时间与签名产物等有效元数据；读取账号失败时不得按空账号继续覆盖。
- 每日续签队列持久化单项完成状态；部分失败后的下一次运行只重试未完成项，不重复成功项。
- 通知调度使用递增 generation 校验，旧的异步授权或调度结果不能覆盖用户最新开关意图。

## 5. 前端交互与导航

- 签名进度和批量续签页面在运行中提供取消按钮；取消请求后按钮禁用并显示等待状态。
- 详情页发起操作时，先关闭详情 sheet，再由父层在 `onDismiss` 中呈现下一 sheet，消除同一事务内的展示竞争。
- 修复安装错误恢复动作：先保留恢复路由，再关闭错误提示，确保“重试/去设置”确实可达。
- 统一设置页的类型化导航，不混用 value-based 与 closure-based destination。
- 移除重复 `.task`/`.onAppear` 强制加载，并在 ViewModel 增加 single-flight 保护。
- 列表改用 `LazyVStack` 和稳定标识；图片解码移出 View body，并使用主线程受控缓存。
- 导入、账号选择、Bundle ID 编辑和进度页面使用可滚动内容及自适应 detent，避免小屏、横屏和大字体截断。

## 6. 主题与视觉规范

- 删除根视图的 `.preferredColorScheme(.dark)`。
- `SealBackdrop` 和所有卡片、分隔线、文字、遮罩改用语义颜色；不在业务视图直接写死黑色或仅适用于深色的白色透明度。
- 浅色主题：蓝灰背景、白色或轻微蓝灰卡片、深蓝灰正文、高对比度蓝色强调。
- 深色主题：深海军蓝背景、略亮海军蓝卡片、接近白色正文；不使用纯黑作为全局背景。
- 保留现有品牌蓝与玻璃层次，但确保浅色模式下边界、阴影和材料可辨识。

## 7. 无障碍与自适应布局

- 固定字号替换为语义字体或可缩放字体；标识符等确需等宽的文字仍随 Dynamic Type 缩放。
- 图标按钮补齐可读 label、hint 和至少 44×44 pt 触控区域。
- 选中账号添加 VoiceOver selected trait；装饰图片从可访问性树隐藏。
- 应用行朗读展示名称和实际可理解状态，不朗读内部原始 Bundle ID 代替名称。
- 开放项目横屏方向，并让主要 sheet、表单和列表在紧凑高度下仍可滚动操作。

## 8. 错误处理原则

- 用户主动取消、系统拒绝、网络失败、Portal 策略冲突和本地事务失败使用不同错误类型与文案。
- 任何远端破坏性动作都要求用户明确触发。
- 错误恢复动作必须是幂等的；重试前先确认文件事务、操作 lease 和临时监听已清理。
- UI 不因取消显示红色失败告警，但会保留未完成状态供再次执行。

## 9. 测试与验收

### 单元测试

- 证书上限不触发 revoke。
- 默认 Bundle ID 使用 `.seal`，已有 Portal 配置只合并不覆盖。
- 多团队登录必须显式选择并持久化选择。
- 导入数据库失败后旧 IPA 完整恢复。
- 自签替换保留账号与安装元数据。
- 清理不会删除正在签名或曾安装的应用。
- 取消传播为 cancellation，且操作资源被释放。
- 部分续签失败后只重试失败项。
- 通知旧请求不能覆盖新意图。
- 孤儿恢复状态不是 installed。

### UI 与静态验收

- 浅色、深色均截图检查，背景无纯黑且对比度足够。
- 小屏、横屏和最大 Dynamic Type 下关键按钮可达。
- VoiceOver 检查应用行、账号选择和所有图标按钮。
- 验证详情到签名 sheet、安装失败恢复、取消和通知开关快速切换。
- 更新失效 UI 测试文案与标识符。

### CI

- 工作流调用项目测试脚本，而不仅是生成工程或构建。
- 本地环境若缺少 Xcode，只执行可运行的静态校验；完整 Swift 编译和 iOS 测试由 macOS CI 完成。

## 10. 分阶段交付顺序

1. 测试基线与 CI 修正。
2. 配对安全、Portal 安全和团队选择。
3. 文件事务、恢复逻辑和操作协调。
4. 签名/安装/续签状态机与取消。
5. 导航、sheet、加载竞态和性能。
6. 自适应主题、无障碍、Dynamic Type 与横屏。
7. 全量回归、静态审查和验收说明更新。

每一阶段都要求新增测试先失败、实现后通过，并在进入下一阶段前检查对既有数据与行为的兼容性。
