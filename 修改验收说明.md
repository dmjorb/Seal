# Seal 最终源码修改验收说明

本说明对应用户确认的 **14 个问题 + 1 个使用教程调整项**。本次修改直接落在签名、续签、Apple Portal、证书、配对、通知、安装通道、启动状态机和 SwiftUI 容器代码中，不是仅修改文案或隐藏控件。

## 1. Apple ID 刷新不再刷新描述文件

- `ApplePortalInventoryService` 增加 `FetchScope`，App ID 与证书分别读取。
- App ID 刷新仅调用 Apple App ID 列表接口，不读取、生成或刷新 provisioning profile。
- `SettingsViewModel` 的 App ID 刷新和证书刷新使用独立 scope，并只合并本地缓存的另一类数据。
- 描述文件仍只在真实签名/续签链路中生成并写回。

主要文件：
- `Seal/Infrastructure/Signing/ApplePortalInventoryService.swift`
- `Seal/Features/Settings/SettingsViewModel.swift`
- `Seal/Features/Settings/AppleAccountDetailView.swift`

## 2. Seal 自身续签自动返回主屏幕

- 安装通道增加 `isSelfReplacement`，普通 App 与 Seal 自替换走不同安装收尾。
- Seal 到达安装阶段后先保存真实签名结果，再启动系统安装请求并自动返回主屏幕。
- 手动续签与每日自动续签共用同一底层流程，没有额外确认弹窗或按钮。

主要文件：
- `Seal/Core/Installation/InstallChannel.swift`
- `Seal/Core/Installation/SelfReplacementController.swift`
- `Seal/Infrastructure/Installation/MinimuxerInstallChannel.swift`
- `Seal/Core/Signing/SigningCoordinator.swift`

## 3. 打开 Seal 后默认轻量检查

- 启动和重新进入前台时自动执行本地轻量检查，无前端开关。
- 读取 Seal 当前 embedded.mobileprovision、已安装 App 本地记录、配对文件和 LocalDevVPN/安装通道状态，并刷新首页数据。
- 该路径不刷新 Apple ID/App ID/描述文件，不创建证书，也不计算“当前时间 + 7 天”。

主要文件：
- `Seal/App/RootTabView.swift`
- `Seal/Features/Apps/AppsViewModel.swift`
- `Seal/Features/Settings/SettingsViewModel.swift`
- `Seal/Core/Renewal/SelfAppRegistrar.swift`

- 状态文案为“可用”。
- 时间显示精确到分钟：`yyyy-MM-dd HH:mm`。
- 不使用刷新时间、描述文件时间或本地加 7 天。

- 提醒固定为到期前 24 小时。
- 通知计划中 Seal 排在最高优先级。
- Seal 详情可发起“本周期一起续签”，按 Seal 的真实到期时间筛选同周期普通 App。
- 批量队列始终普通 App 在前、Seal 最后。
- 进行中界面仅显示“正在续签”和 `x / y`，结束后显示成功/失败结果。

## 7. 有效期只使用真实来源

- Seal 自身：读取 App 包内 `embedded.mobileprovision`。
- 普通 App：签名后解析实际嵌入的描述文件并写入到期时间。
- App ID：使用 Apple 返回的 App ID 到期时间。
- 通知与剩余天数均消费已保存的真实 `expiryDate`，未加入本地 +7 天推算。

主要文件：
- `Seal/Infrastructure/Renewal/ProvisioningProfileReader.swift`
- `Seal/Core/Renewal/SelfAppRegistrar.swift`
- `Seal/Core/Signing/SigningCoordinator.swift`
- `Seal/Infrastructure/Signing/ApplePortalInventoryService.swift`

## 8. 待签名与已安装 App 的身份规则分离

- 待签名 IPA：Bundle ID 可编辑，Apple ID 可选择。
- 已安装 App/Seal：Bundle ID 与 Apple ID 不可修改。
- 续签强制使用上次保存的 Apple ID 和 Team；缺少记录或不匹配时直接阻止。
- 证书使用该账号当前本机可用的开发证书，外部传入的其他证书 Serial 不能覆盖。
- 不读取“我的”页当前选中的 Apple ID 作为续签默认账号。

主要文件：
- `Seal/Core/Signing/BundleIDPolicy.swift`
- `Seal/Core/Signing/SigningCertificateSelectionPolicy.swift`
- `Seal/Features/Apps/AppsViewModel.swift`
- `Seal/Features/Apps/AppSigningSheet.swift`
- `Seal/Core/Apps/AppRecord.swift`

## 9. 证书自动处理

- 没有本机可用证书时自动创建并立即继续当前签名任务。
- 证书满额时重新拉取远端证书列表，保护本机仍有有效私钥的证书，撤销其余旧证书/无本机私钥证书，并在每次撤销后立即重试创建。
- 新证书创建后重新拉取 Apple 证书列表，绑定私钥、生成 P12、写入 Keychain 和账号状态，然后继续当前任务。
- 账号残留 Serial 但 Keychain 没有匹配 P12/私钥时会先清除失效绑定，避免把无私钥证书误判为受保护证书。
- 前端阶段统一为：处理证书、创建 App ID、生成描述文件、签名、安装。

主要文件：
- `Seal/Infrastructure/Signing/ApplePortalSigningService.swift`
- `Seal/Core/Signing/SigningCoordinator.swift`
- `Seal/Core/Signing/SigningStage.swift`
- `Seal/Features/Apps/SigningProgressView.swift`

## 12. 权限与真实系统通知

- 首次启动只触发本地网络权限路径。
- 通知权限仅在用户打开“到期前 24 小时提醒”时请求。
- 使用 `UNUserNotificationCenter` 创建本地系统通知。
- 提醒固定按真实到期时间倒推 24 小时；若该准确触发点已过去，不将它平移成错误时间。
- 每次重排先取消旧的 Seal 到期通知，再按最新真实到期时间创建。
- 关闭开关或启用失败时取消全部 Seal 到期通知。
- 文案按 Seal/普通 App 分开。

主要文件：
- `Seal/Infrastructure/Notifications/ExpiryNotificationScheduler.swift`
- `Seal/Core/Notifications/ExpiryNotificationPlanner.swift`
- `Seal/Core/Notifications/NotificationPreferences.swift`

- 卡片/分组/按钮/状态色统一使用 Seal 设计系统颜色。

## 14. 每天第一次打开自动续签全部

- 使用 UserDefaults 键 `lastAutoRenewDate` 保存本地自然日键值。
- 使用 `Calendar.current` 的年/月/日定义本地自然日。
- `autoRenewInProgress` 与现有签名/批量任务锁共同防止并发重复执行。
- 当天全部成功后不再执行；失败时不标记当天成功，可再次尝试。
- 普通 App 先续签，Seal 最后。
- Seal 自替换可能使当前进程暂停，因此额外保存待确认状态；下次启动只有检测到新 embedded.mobileprovision 到期时间确实前移，才确认当天任务成功。
- 若普通 App 已失败，不会因为 Seal 自身安装成功而误把整批任务标记为成功。

主要文件：
- `Seal/Core/Renewal/DailyAutoRenewStateStore.swift`
- `Seal/Features/Apps/AppsViewModel.swift`
- `Seal/App/RootTabView.swift`
- `Seal/Core/Renewal/RefreshPlanner.swift`
- `Seal/Core/Renewal/RenewalCoordinator.swift`

## 验证边界

当前执行环境为 Linux，没有 Xcode、iOS SDK、Apple Developer Portal 真账号环境和连接的 iPhone。因此无法在这里完成 Xcode 全量编译、Apple 服务端实签、LocalDevVPN 真机安装或 Seal 自替换安装的设备级验证。源码层修改、静态检查、可运行核心逻辑检查均已完成；真机最终验收仍需在 macOS + Xcode + 当前 iPhone 环境执行。
